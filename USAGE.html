<<showtoc>>
!Function Access
If first parameter starts with "_", script will internally execute all given parameters, literally as "$@". Thus, all internal functions can be unit tested on the fly, or directly used as part of other applications.
//{{{
./ubiquitous_bash.sh _false		#Returns 1.
//}}}
Exit status is preserved.

!Import
Script recognizes several import modes, identified by first parameter, which is shifted out if found.
//{{{
#"--profile"
#"--parent", "--return", "--devenv"
#"--call", "--script", "--shell", "", "--bypass"
//}}}
If imported, script will generally "return" before reaching "main()". Using <nowiki>"--bypass"</nowiki> will force normal exit and allow main to be reached, even if script is not imported.

Safety checks (including "_safeRMR") rely upon an accurate "$scriptAbsoluteLocation" setting, which cannot be obtained automatically if launched from a login shell. Consequently, all import modes require external information from parent script or user.
//{{{
export importScriptLocation="$PWD"/ubiquitous_bash.sh
export importScriptFolder="$PWD"
. "$importScriptLocation" --call
//}}}

Script location may be obtained by _getAbsoluteLocation, etc, from parent script. 
//{{{
export importScriptLocation="$getScriptAbsoluteFolder"/ubiquitous_bash.sh
export importScriptFolder="$getScriptAbsoluteFolder"
. "$importScriptLocation" --script
//}}}

Import behavior may be manually tested under a subshell of a login shell.
//{{{
rm ~/.ubcore/userlog/* >/dev/null 2>&1 ; bash -c 'export importScriptLocation="'"$PWD"'"/ubiquitous_bash.sh ; export importScriptFolder="'"$PWD"'" ; . ./ubiquitous_bash.sh' ; echo "$?" ; cat ~/.ubcore/userlog/* ; rm ~/.ubcore/userlog/*
//}}}

!Bash Profile
An import directive for Ubiquitous Bash functions and global variables can be automatically installed to a user's ".bashrc" file.
//{{{
./ubiquitous_bash.sh _setupUbiquitous
//}}}

Once installed, the shell environment's function set may be updated from git by calling the same function, now available directly as a normal shell command.
//{{{
_setupUbiquitous
//}}}

Certain functions will be automatically executed, for example, to provide a more informative and separative bash prompt. All are documented as commands in the "_importShortcuts" function.
//{{{
export importScriptLocation="$PWD"/ubiquitous_bash.sh
export importScriptFolder="$PWD"
. "$importScriptLocation" _importShortcuts --call
//}}}

WARNING
Functions not documented as shortcuts or end-user (eg. virtualization) commands, could create a mess if run directly without their prerequisite _start/_stop/_open/_close/_failExec . Eventually, all such functions will be verified to have completely independent environment sanity checks. In the meantime, this is a developer feature.

!Dependency Checking

	if (! evt)
		var evt = window.event;

	var target = resolveTarget(evt);
	var tiddler = story.findContainingTiddler(target);

window.scrollToHeading = function(title, anchorName, evt) {
	var tiddler = null;

	if (! evt)
		var evt = window.event;

	if (title) {
		story.displayTiddler(store.getTiddler(title), title, null, false);
		tiddler = document.getElementById(story.idPrefix + title);
	} else {
		var target = resolveTarget(evt);
		tiddler = story.findContainingTiddler(target);
	}

	if (tiddler == null)
		return false;
	
	var children1 = tiddler.getElementsByTagName("h1");
	var children2 = tiddler.getElementsByTagName("h2");
	var children3 = tiddler.getElementsByTagName("h3");
	var children4 = tiddler.getElementsByTagName("h4");
	var children5 = tiddler.getElementsByTagName("h5");

	var children = new Array();
	children = children.concat(children1, children2, children3, children4, children5);

	for (var i = 0; i < children.length; i++) {
		for (var j = 0; j < children[i].length; j++) {
			var heading = children[i][j].innerHTML;

			// Remove all HTML tags
			while (heading.indexOf("<") >= 0) {
				heading = heading.substring(0, heading.indexOf("<")) + heading.substring(heading.indexOf(">") + 1);
			}

			// Cut off the code added in showtoc for TOP
			heading = heading.substr(0, heading.length-6);

			if (heading == anchorName) {
				var y = findPosY(children[i][j]);
				window.scrollTo(0,y);
				return false;
			}
		}
	}
	return false
};

!Pitch
Have you ever wanted to ship out an application in a virtual machine, and have it just work with files on the host? Or maybe just wanted to have proper temporary folders based on a session id? Or perhaps correctly recurse through a shell script's functions, by executing that script with a function name, from that same script? Or even just access files in the same directory as a script?

Ubiquitous Bash is the answer to these problems, and more. Written in Bash, the framework has been tested under Linux distributions, and partially tested under Cygwin. Test reports for BSD/Mac welcome as well, but try with careful awareness that safeRMR may not work as expected there. Supporting multiple virtualization backends (eg. ~VirtualBox, Docker) targeting the same image, changes to the underlying technology will not let you down. With over 4000 lines of modular shell code, and over 400 externally accessible functions, like '_getAbsoluteLocation', '_discoverResource', '_timeout', '_daemon', '_start', '_stop', and '_qemu', this is a huge package of functionality you can't miss!

From a Linux host, passing the full "./ubiquitous_bash.sh _test" suite, running these commands demonstrates correct operation of some of the most unique and sophisticated components. Patches will likely be tested against this command set before acceptance.

As "a dedicated monolithic kernel for (meta)applications", use of global variables is encouraged, subject to four conditions.
# Global variables must be set on every run, none may transfer to a new instance. If script is imported from within itself, all global variables must be re-set (even if to blank values) by the new instance.
# Indefinite recursion may not reuse an existing session, may not import script from within itself. Thus, global variables which extend existing global variables (eg. "$PATH":"$scriptBin" ), may not expand indefinitely.
# Session ID, set to random value on every run, must be included in path to all resources (eg. safeTmp="$scriptAbsoluteFolder"/w_"$sessionid" ) which need not be more than temporary.
# Fundamental operators (eg. open, close) are to be identified prior to implementation and grouped together. Recognition of fundamental operators should guide a rigorous separation of concerns, and names should reflect these origins.

Variables pointing to essential resources used by most applications are enumerated here. A succinct list of all global variables and their exact definitions is provided at "structure/globalvars.sh".

!Session ID
* "$sessionid"
A 18 character alphanumeric random string generated once each time the script is run. All functions called from within the script will see the same session identifier value.

!Script Location
* "$scriptAbsoluteLocation"
Expanded path for the script's location, evading many pitfalls, especially regarding symbolic links, which might otherwise compromise the use of "$0".

* "$scriptAbsoluteFolder"
Folder in which script resides, ultimately derived from "$0". Used to locate neighboring resources and safe places for temporary files.

!PWD
* "$outerPWD"
Records the current directory, so a function can perform 'cd "$outerPWD" ' before returning control elsewhere.

!Persistent Directories
* "$scriptLocal"
Data specific to this host. Typically server configuration files, virtual machine images, daemon PID files, and log file backups.

* "$permaLog"
Set to match "$scriptLocal".

* "$scriptBin"
Directory for executable files needed (and often compiled) by script. Reference "$scriptBin" directly to ensure the correct executable is used. Nevertheless appended to system "PATH" variable.

!Temporary Directories
* "$safeTmp"
All purpose temporary directory, in same folder as script, starting with "w_" and ending with unique session id. While the script is running, the contents of this folder, including log files, may be useful to show its current state. If this folder somehow persists after script is no longer running, it is almost certainly obsolete, and its contents may show the cause for such an unusual failure.

* "$logTmp"
Log directory under "$safeTmp". Log files written here do not need unique names, but can be backed up to a permanent location if _preserveLog is called by _stop function .

!Special
* "$objectName"
Basename of the directory containing script. Typically used to cause the script to call a function based on the name of the folder it resides in. As an example, used by CoreAutoSSH project to automatically launch VNC clients, through SSH tunnels, with a script placed under a folder named, according to the for the target machine host/domain name.

!Recursion
Due to complete thread safety, the script may always call itself, providing separate temporary resources. Built-in "$0" is expanded to "$scriptAbsoluteLocation", then used as the executable name.
//{{{
"$scriptAbsoluteLocation" _buildSequence
//}}}
Functions using "_start" / "_stop" should be recursively called from within the script in this manner to prevent interference in the parent's processes.

!Instancing
Temporary files/directories, cleanup, crash handling, and configuration file reading/writing, are handled by functions under "structure/localenv.sh".

!!Example
//{{{
_start

/bin/true || _stop 1

_stop
//}}}

!!_start
Creates temporary directories, reads configuration file, writes out temporary pid file.

!!_stop [1]
Copies diagnostic log files, removes temporary files/directories, kills dameonized process (if running). Optionally may save chosen variables to configuration file. Reports 0 or parameter [1] as exit status.

!!_stop_emergency
Accelerated shutdown procedure intended to free up the host filesystem if at all possible. Unmounts shared resources (ie. disk images) as fast as possible before handing off to normal _stop with error status. Typically called upon SIGINT, SIGTERM, or during system shutdown.

!!_collect
Gathers command/variable scripts from other (ie. yaml) file types (ie. ~AppImage recipes). Rarely implemented.

!Locking
Multiple processes cannot update the same data in shared files, or severe corruption results. Generic locking handlers are provided to prevent these situations, and include support for atomicity as well as reboot detection. Implementation code is at "structure/localenv.sh".

Instanced resources should always be used instead of locking shared resources if at all possible. One unique exception is the direct involvement of scarce cognition (eg. human resources), where the prevalence of too many versions may lead to confusion.

Keep in mind modern applications and operating systems impose rarely encountered, but dangerous pitfalls, so documentation of this framework may involve some oversimplifications. Deliberately exploring the actual code with careful thought experiments is recommended throughout the development process.

!_readLocked ["$scriptLocal"/lockFile]
Checks whether a lock file exists, and has not been rendered obsolete by reboot. If obsolete, old lock file is removed.

!_createLocked ["$scriptLocal"/lockFile]
Writes lock file, using a move and check existence operation, to catch events which did not occur atomically.

!_resetLocks
Removes obsolete standard lock files, using _readLocked .

!!_open [_waitOpen] [_writeData]
Checks standard lock files, including "$lock_open", and "$lock_opening". If open, returns true (ie. 0). If already opening, waits for first parameter [_waitOpen] to return true, indicating any required mounting operations are finished, then returns true. If not already open, and not already opening, sets "$lock_opening", then executes all remaining parameters ["$@"]. If successful, removes "$lock_opening", and finally sets "$lock_open".

!!_close [_waitClose] [_syncData]
Inverse of open operation. Checks standard lock files, including "$lock_open", and "$lock_closing". If not open (ie. already closed), return true. If already closing, waits for first parameter [_waitClose] to return true, indicating any required dismounting operations are finished, then returns true. If not already closed, and not already closing, sets "$lock_closing", then executes all remaining parameters ["$@"]. If successful, removes "$lock_closing", and finally removes "$lock_open".

!Compiling
Ubiquitous Bash is a "statically linked" executable script, compiled by the "compile" script, entirely from submodule text files, containing only closely related functions.
//{{{
./compile.sh
//}}}

Specific versions may be compiled by passing parameters, and overriding functions with "ops".
//{{{
./compile.sh lean lean.sh
//}}}

!Hiearchy
During compilation, submodule text files are searched for, starting with the directory "_prog". Thus, to compile a script which executes program specific code by default, first the text file "structure/entry.sh" would be copied to a new directory as "_prog/entry.sh". Changes to this file would then be placed at the end of "ubiquitous_bash.sh", instead of the original code. In this way, entirely new applications can be constructed, while easy to update with upstream improvements to the Ubiquitous Bash middleware.

!Overloading
After all functions and variables are defined, but before any actions are performed, an "ops" file in script's parent directory will be imported if present. Any function declarations, variable definitions, or code in this file will execute within the context of the parent script (ie. "ubiquitous_bash.sh"). Thus, this is a dynamic method to override precisely just what is needed, at run time. In simple cases, this may simplify software maintenance, or provide infinitely flexible end-user configuration. Automatic generation by graphical programs also managed with an Ubiquitous Bash middleware layer is another possibility.

!Forking
When not developing Ubiquitous Bash itself, it is strongly recommended to import the middleware as a "git submodule" in a new project. This will allow a quick "git pull" to incorporate any changes, which occasionally include additional safety checks.
//{{{
mkdir project
cd project

mkdir ./_lib

git init
git submodule add git@github.com:mirage335/ubiquitous_bash.git ./_lib/ubiquitous_bash
git add -A .
git commit -a -m "first commit"

cp ./_lib/ubiquitous_bash/compile ./
//}}}

Forking can be done automatically with the "fork" script, which will also copy some of the most commonly modified text submodules to a "_prog" subdirectory for editing. This approach is strongly recommended.
//{{{
#Download fork script anywhere.
#Edit devName and projectName variables.
./fork
//}}}

!Command Installation
Symbolic links to user PATH may be added automatically based upon directory name, script name, or find commands to locate other scripts. See "structure/installation.sh", specifically functions _setupCommand and _setupCommands . Overloading these functions with "ops" is best practice.

!Best Practices
!!Anchor and Call
Anchor and Call scripts are shortcuts, searching based on rules, rather than following a single pointer, as with symlinks. Anchor scripts are general purpose links, searching mostly down, passing all parameters. Call scripts may be special purpose, checking hardcoded paths, opening a program at a specific location.

System PATH should only include symlinks to Anchor and Call scripts. Both internally determine their actual path, so copying them to unintended locations results in undefined behavior.

Anchor scripts are typically symlinked to PATH. Anchor scripts must be installed in one of a few locations illustrated by [[Ubiquitous Lab]].
//{{{
./_anchor ./file.txt
"$HOME"/core/installations/ubiquitous_bash/ubiquitous_bash.sh _anchor ./file.txt
//}}}

Call scripts are typically copied to project directories. Call scripts may be installed anywhere. Call scripts may include hardcoded paths, arbitrary parameters, or filesystem specific lock mechanisms.
//{{{
./_call ./file.txt
"$HOME"/core/lab/labName/_index/_call --user-specified ./file.txt
//}}}

!!Specialized Dependencies
A "_tryExec" function is generally used to call a function which checks for dependencies used only by specific submodules. If the relevant test function has not been declared, because its submodule has not been included, because the relevant functionality is not needed in the current project, then the dependency check will be skipped automatically.

Top-level control of meta-applications, referred to as a "lab", is a primary goal for Ubiquitous Bash.

Following an "Ubiquitous Lab" standard layout will allow modules to be added and removed as needed for contingencies or efficiency.

!Conventions
!!Generic Virtual Machines
//{{{
"$HOME"/core/infrastructure/winexevm
"$HOME"/core/infrastructure/nixexevm
//}}}

!Rules
* An anchor searches down, findInfrastructure searches up.
* Strict rules allow anchors to search up in specific cases. An anchor never looks beyond user HOME directory, or more than two directories above, only one directory above unless parent is "_local/_index".
* All programs MUST launch from the lowest subdirectory available, confining data, configuration, FakeHome, and similar resources, to one location - that which will be shared most widely.

* Shortcuts to specific applications should be placed under "core/_index" folders. Such shortcuts should be anchors.
* Shortcuts to launch multiple, possibly connected, applications, should be placed under a "core/_meta" folder, referencing anchors.

* Images of disk operating systems should be placed above the "Mini" lab if possible. Future reassignment should be simple as moving or copying the directory.

!Overloads
Consider setting the following variables, through "ops", for production Ubiquitous Bash repositories.
//{{{
#Enable search if "vm.img" and related files are missing.
export ubVirtImageLocal="false"
//}}}

!Directory
Like any Filesystem Hiearchy Standard, a hierarchical approach allows programs to automatically detect other programs. However, here the goal is to allow neighboring programs to reside in parent or subdirectories. Lightweight "Mini" versions of any lab environment may thus be kept as subdirectories of a parent, which may include larger virtualization images, or backups, not needed in production.
//{{{
labName
	_index
	_local
		_index
	core
		infrastructure
			vm
				vmFirstApp
				vmSecondApp
			lab
				labNameMini
					core
						infrastructure
							vm
								vmSecondApp
							lab
								labOther
//}}}
Here, "vmSecondApp", as included with "labNameMini", might not have the necessary "vm.img" file. Ubiquitous Bash "_findInfrastructure_virt", would locate the "vmSecondApp" script which actually does have the necessary "vm.img" file, sending commands to that script instead. Several contingencies are available, and may be overloaded further, as documented by "virtualization/findInfrastructure_virt.sh".

!Anchor
Applications, virtual machines, and complete labs, may need to be reassigned by moving their directories, as a complete module. Symlinks, pointed at a specific target, might not follow these changes, leading to data not being saved where expected, and subsequent data-loss accidents. Anchor scripts, may be used to guarantee the lowest subdirectory available is used as a source for these programs, providing consistent expectations.

Anchor scripts, by default, search for a named Ubiquitous Bash fork, as well as other anchors, but may be configured only to search for other anchors, as would be appropriate for other labs.

See "_anchor" script for template.

!Images
Any bootable raw image may be used by ~ChRoot, QEMU, or ~VirtualBox. Raw images may be converted for use by Docker and vice versa. See [[Debian]], [[Raspbian]], or [[Docker|DockerCreate]] for built-in creation procedures.

Images may be edited more easily with several shortcuts. Please be aware the _toImage/_fromImage commands are equivalent to "cp -a", and will not overwrite or delete destination contents.
//{{{
./ubiquitous_bash.sh _toImage ./.arduino /etc/skel
./ubiquitous_bash.sh _toImage ./.arduino /home/user

./ubiquitous_bash.sh _fromImage /home/user/.arduino ./
./ubiquitous_bash.sh _fromImage /etc/skel/.arduino ./

./ubiquitous_bash.sh _toImage ./exim4 /etc/exim4
./ubiquitous_bash.sh _fromImage ./exim4 /etc/exim4

./ubiquitous_bash.sh _closeChRoot
//}}}

Some directories have special significance within the ubiquitous_bash framework. In particular, the /etc/skel directory is used by both ~ChRoot and Docker components to build the home folder for the temporary "ubvrtusr". Similarly, /home/user is directly used by QEMU and ~VirtualBox. For ~ChRoot, only the relevant "_root*" command can also make persistent changes. For QEMU, ~VirtualBox, and Docker, relevant "_edit*" commands may also make persistent changes.

!<nowiki>FakeHome</nowiki>
Many programs inappropriately assume the user's "home" folder is the appropriate place for configurations which might actually need to be different for different use cases. Overriding the "$HOME" environment variable, more rarely other variables as well, can resolve the resulting conflicts. ~FakeHome is a feature of ubiquitous bash which goes a few steps further depending on which function is used, what environment variables are set, what overrides are present in "ops", and whether setupUbiquitous has hooked the user's actual ".bashrc". For example, it is possible to automatically drop the redirection when a later subshell is launched.

End user FakeHome functions documented here are also intended as documentation themselves. Application developers needing more granular control of the fakeHome environment may use their simple sequences as examples. See "virtualization/fakehome/fakehomeuser.sh" .

!!Create
Automatically adds typical shared directories, such as "~/Downloads" to ~FakeHome environment, by symlink.
//{{{
_createfakeHome
//}}}

!!Root
Fake home directory is set to "_local".
//{{{
_editFakeHome firefox --no-remote
//}}}

!!User
Temporary fake home directory, based upon the contents of "_local".
//{{{
_userFakeHome firefox --no-remote
//}}}

!!Mem
Same as user, but on a ramdisk. Requires effective root.
//{{{
_memFakeHome firefox --no-remote
//}}}

!!Short
Some applications (eg. gpg) will not operate correctly if the "$HOME" variable contains a long path. As a workaround, a directory in "/tmp" can be used instead of the default "_local/h".
//{{{
_userShortHome
//}}}

Because symlinks will generally be deferenced if used as "$HOME", this workaround necessitates symlinking specific application's configuration directories.

Override, or implement, the example "_install_fakeHome_app" in "ops", "core.sh", or elsewhere, as appropriate for specific applications.
//{{{
_install_fakeHome_app() {
	#_link_fakeHome "$scriptLib"/app/.app ".app"
	true
}
//}}}

As a useful convention, consider using the "_local/app" default for persistent app configuration directories, unless a specific type of application for integration with meta-applications is to be denoted (ie. blockchain).

export keepFakeHome="false"

//}}}
